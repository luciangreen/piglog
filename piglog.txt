There are several computational models in which this is possible. The important distinction is that the predicate is partially evaluated rather than fully executed. Here are ways Piglog could support each case.

Some variables are known
Example:

distance(X,Y,D).

Suppose only X is known.

Instead of waiting for Y, Piglog could compile the predicate into stages:

Stage 1: Compute everything depending only on X.
Stage 2: Suspend.
Stage 3: Resume when Y arrives.

Much like currying or partial application in functional programming, except for logic predicates.

Constraints are known
Suppose:

X > 0,
Y > X,
f(X,Y,Z).

The predicate can already infer things like:

Y > 0
impossible branches
valid search intervals

before either variable has a concrete value.

Constraint Logic Programming already does something similar by propagating constraints without assigning values.

Piglog could let every predicate publish:

required constraints
derived constraints

allowing constraint propagation to occur concurrently.

Probabilities are known
Suppose a learned model predicts

X = cat 0.82
X = dog 0.15
other 0.03

Instead of waiting,

Piglog may begin evaluating

animal(cat)

immediately,

while evaluating

animal(dog)

at lower priority.

If later evidence changes the probabilities, unfinished work can simply be discarded.

This resembles speculative execution used in modern CPUs.

Invariants are known
Many predicates have parts independent of particular values.

Example:

sort(Input,Output).

Before seeing Input, Piglog already knows

length(Output)=length(Input)
Output is ordered
Output is a permutation

Those facts can propagate through the graph immediately.

Similarly,

matrix_multiply(A,B,C)

may immediately infer

rows(C)=rows(A)
cols(C)=cols(B)

before any multiplication begins.

Possible values are known
Suppose

colour(red).
colour(green).
colour(blue).

Instead of

red
then green
then blue

Piglog could launch three workers simultaneously.

Later predicates receive a stream of candidate bindings instead of waiting for one answer at a time.

A sixth possibility: symbolic execution

Instead of concrete values,

X = ?

Piglog keeps

X = symbolic

For

plus(X,2,Y).

it computes

Y = X+2

without needing X.

Only when X becomes

5

does

Y

collapse to

7

Many theorem provers and SMT solvers already operate this way.

A seventh possibility: dependency scheduling

This may be closest to your Piglog concept.

Instead of scheduling predicates by textual order,

a,
b,
c,
d.

Piglog schedules according to dependency readiness.

    a
  / | \
 b  c  e
  \ | /
    d
The scheduler asks

"Which predicates have enough information to make progress?"

rather than

"Which predicate is next?"

Every predicate exposes:

required inputs
optional inputs
invariants
partial outputs

As soon as enough information is available, it runs.

This leads to a possible Piglog execution model with explicit readiness levels:

Readiness Information available Action
0 None Idle
1 Types/invariants Compile or specialise the predicate
2 Constraints Propagate constraints, prune impossible branches
3 Probability distribution Speculatively execute likely branches
4 Partial variable bindings Partially evaluate the predicate
5 Complete bindings Produce definitive results

Instead of treating predicates as requiring all inputs before starting, Piglog would allow them to make progress whenever they reach a sufficient readiness level. This turns execution into a form of dataflow scheduling, where information—whether exact values, constraints, or probabilities—flows through the predicate network and progressively enables more computation.

Here is a stronger Piglog Prolog program requirement:

Piglog should not “run a predicate early” by pretending unknown variables are known. It should split every predicate into readiness modes.

predicate_state(Predicate, Readiness, Inputs, Outputs).

A predicate may run when it can produce any useful information, not only final answers.

Example:

p(X,Y,Z) :-
q(X,Y),
r(Y,Z),
s(X,Z).

Piglog converts this into a dependency network:

X known? -> q can specialise
Y known? -> r can run
X,Z constrained? -> s can prune

Each predicate has multiple entry points:

q_full(X,Y,Result). % all variables known
q_partial_x(X,Constraints). % only X known
q_constraints(C0,C1). % constraints known
q_probs(P0,P1). % probabilities known
q_invariants(I). % always true facts
q_candidates(Candidates). % possible values known

So the runtime asks:

What can this predicate do now?

not:

Can this predicate fully finish now?

A practical requirement could be:

:- pig_predicate q/2.

pig_mode(q/2, full, [ground, ground], produces(result)).
pig_mode(q/2, partial_x, [ground, var], produces(constraints)).
pig_mode(q/2, constraints, [constraintset], produces(pruning)).
pig_mode(q/2, probable, [distribution], produces(speculative_jobs)).
pig_mode(q/2, invariant, [], produces(invariants)).

Then Piglog schedules jobs like:

ready(Job) :-
job(Job, Predicate, Mode),
mode_requirements_satisfied(Predicate, Mode).

A Piglog predicate must declare what it can compute at each information level.

Pigdetlog runs a predicate if it knows the variables to a high probability, based on their splice data

That is an interesting extension of your Pigdetlog idea. In effect, you are proposing that execution order should be guided not just by syntactic dependencies (which variables are already bound), but also by predicted bindings derived from splice data.

One way to formalise it is:

Pigdetlog executes a predicate when the probability that its required variables are correct, as inferred from splice data, exceeds a configurable confidence threshold.

This differs from ordinary Prolog and your earlier Detlog ideas:

System When is a predicate executed?
Prolog Only when variables are sufficiently instantiated according to program order.
Detlog When deterministic loops and splice operations make execution possible.
Pigdetlog When splice analysis predicts the missing variables with sufficiently high confidence.

For example, suppose you have

person(Name, Age, City).
income(Age, Salary).
tax(Salary, Tax).

Traditional execution requires

person(Name, Age, City),
income(Age, Salary),
tax(Salary, Tax).

Pigdetlog might observe from previous splice data that

Age ≈ 35 (98%)
Salary ≈ 120000 (94%)

and therefore begin evaluating

tax(120000, Tax)

before income/2 has actually produced Salary. Later, when the actual splice arrives:

if prediction matches, computation is retained;
if prediction differs, the speculative result is discarded and recomputed.

The splice data effectively becomes a probabilistic cache.

Internally, Pigdetlog might represent variables as

Age =
{
value:35,
probability:0.98,
source:splice
}

and predicates could declare confidence requirements, for example

:- confidence(tax/2, 0.90).
:- confidence(expensive_search/3, 0.99).

Then the scheduler could operate like

for each predicate P:
estimate probabilities of missing variables
if minimum confidence ≥ threshold(P)
execute P
else
wait for more splice information

This makes Pigdetlog resemble speculative execution in CPUs, but at the logical level. Instead of speculating on branches, it speculates on logical variable values using splice information. It also has similarities to Bayesian inference and probabilistic logic programming, but the probabilities are used primarily for execution scheduling rather than changing the semantics of logical truth.

Pigdetlog schedules predicates opportunistically whenever splice data predicts their inputs with high confidence, allowing expensive predicates to start earlier and potentially improving throughput on parallel hardware.
Splice probability could be calculated from probability within a list and splice. Separately, Pigdetlog could schedule expensive predicates in regular available slots.

Expensive predicate scheduling Separately, expensive predicates do not need to run immediately. They can be placed into regular execution slots:
cheap predicates: run now

expensive predicates: run in scheduled slots

So Pigdetlog becomes:

Detlog = deterministic splicing
Piglog = simultaneous predicate perspective

A nice rule would be:

Run predicate P when:

confidence(inputs(P)) >= threshold(P)
and
cost(P) fits current available slot

For example:

:- cost(tax/2, cheap).

:- threshold(expensive_search/3, 0.98).
:- cost(expensive_search/3, expensive).

So Pigdetlog does not merely ask:

“Are the variables known?”

It asks:

“Are they probable enough, and is this the right execution slot?”

That gives you a clearer architecture: probability decides readiness; scheduling decides timing.

An example of a definite value is a memoised value, such as an initial loop value, or the Gaussian formula result of having run a loop, or the tabled or probable pattern-suggested subterm indexing looping (loop) value.

Yes. Those examples point toward a useful distinction in Piglog between definite, derived, and predicted values. I would refine the terminology as follows.

Definite values

These are values that Piglog can treat as established without further search.

Examples include:

Input values supplied by the caller.
Memoised values from previous executions.
Initial loop values, such as the current loop index or accumulator.
Tabled values that have already been computed.
Results of deterministic algorithms, such as a Gaussian elimination result once the elimination has completed.
Constants and invariants.

These values have confidence 1.0 and need not be recomputed unless invalidated.

Derived values

These are not stored initially but can be obtained deterministically from definite values.

Examples include:

A Gaussian elimination solution computed from known coefficients.
A value obtained by subterm indexing.
A value produced by deterministic looping.
A splice result assembled from definite branches.

Although these are computed, once obtained they become definite.

Predicted values

These are values that have not yet been proved but are sufficiently likely to justify speculative execution.

Examples include:

Pattern-suggested subterm-indexing candidates.
Frequently occurring loop values.
Values inferred from previous executions.
Values ranked highly by dependency analysis.
Probable splice candidates.

These carry a probability or confidence score and should be verified before being committed as definite.

For Piglog, execution priority could therefore become:

Use definite values.
Compute inexpensive derived values.
Predict high-confidence values if needed to enable later runnable sections.
Execute sections that have become runnable.
Verify predictions and update memoisation or tables.

This fits well with your idea that, after PLOP has removed unnecessary variable pathways, Piglog's primary optimisation is no longer exhaustive parallel execution but efficient discovery of the values that unlock the largest number of downstream algorithm sections. In that sense, the dependency graph is better viewed as a value-dependency graph than a traditional predicate call graph.

dependency and cancellation records;

1. Replace batch parallelism with wavefront execution

Ordinary parallel execution often looks like:

start A, B, C
wait for A, B, C
start D, E

Piglog should instead use:

start A, B, C
A finishes → immediately start D
C fails → cancel everything dependent on C
D produces partial output → immediately start E
B continues independently

There is no global barrier. The active frontier of computation continually moves through the dependency graph.

This is better described as wavefront parallelism or dataflow execution.

2. Split predicates by logical milestones, not only predicate boundaries

A long predicate should not occupy one processor until it is entirely finished. It can expose intermediate milestones:

    parse(Input, Tree),
    classify(Tree, Type),
    transform(Tree, Type, Output).

partial Tree becomes available
classification starts on complete subtrees
transformation starts on classified subtrees
parsing continues

Thus parallelism can stack within predicates, not only between predicates.

A predicate should be compiled into resumable segments:

segment 1: establish structure
segment 2: produce first usable binding
segment 3: refine constraints
segment 4: produce final result

Later work may start after segment 2 instead of waiting for segment 4.

cheap contradiction test
        ↓

4. Use asymmetric parallelism

Not all processors should perform the same kind of work. Piglog can divide them into temporary roles:

producer processors generate candidate bindings;
validator processors test constraints and likely failures;
continuation processors execute newly unlocked predicates;
speculation processors explore probable future paths;
commit processors preserve Prolog-visible result ordering;
cleanup processors cancel branches and reclaim state.

These roles need not be fixed to physical cores. They are scheduling classes.

This prevents expensive producers from filling the system with candidates faster than validators can reject them.

5. Parallelise alternatives and dependencies differently

Piglog should distinguish:

AND-parallelism

Independent goals required by the same solution:

    a(X),
    b(X),
    c(X).

OR-parallelism

Alternative clauses or candidate values:

p(X) :- a(X).

Run alternatives concurrently, but control how answers are committed.

Pipeline parallelism

A stream of results moves through stages:

generate → test → transform → collect

Run a later stage before all certainty is available, using versioned assumptions.

Failure parallelism

Run cheap rejection tests concurrently with candidate production.

Piglog’s novelty should be the coordinated combination of all five, rather than one isolated form of parallelism.
