Piglog qa

1. Project scope
Is Piglog a new Prolog language, a source-to-source compiler, an interpreter, or all three?

Should the first implementation target SWI-Prolog only, or be portable to Scryer, SICStus, etc.?

It should be a converter that works in the SWI-Prolog interpreter.

Is Piglog intended to preserve standard Prolog semantics except for execution strategy?

Yes, there should be a Piglog command that runs the algorithm with an option to trace it (and sequential, speculative or adaptive options). Ideally, the code that runs the algorithm should be optionally output with the algorithm and saved for faster running. For example, in sequential mode, as soon as a variable becomes available, partitions depending on it should run in parallel or be queued. There should be commands to lock off Piglog in cases where password walls shouldn't give away secrets before passwords are entered or answers to questions shouldn't be given before the user has answered or requested the answer.
